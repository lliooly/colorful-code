# Agent 后端 API 文档

**适用前端：** `apps/web`

**后端入口：** `apps/server`

**默认 Base URL：** `http://localhost:3001`

**Content-Type：** 除 SSE 和 `DELETE` 外，所有请求均使用 `application/json`。

**鉴权：** 当前接口未接入登录态或 Token 鉴权。

---

## 1. 接口总览

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/health` | 服务健康检查 |
| `GET` | `/health/tools` | 内置工具健康检查 |
| `POST` | `/sessions` | 创建新会话 |
| `POST` | `/sessions/:id/restore` | 从持久化快照恢复会话 |
| `POST` | `/sessions/:id/messages` | 向会话发送用户消息 |
| `POST` | `/sessions/:id/control` | 发送运行中控制消息 |
| `GET` | `/sessions/:id/events` | 订阅会话 SSE 事件流 |
| `GET` | `/sessions/:id/snapshot` | 读取会话持久化快照 |
| `GET` | `/sessions/:id/audit` | 读取权限审计记录 |
| `GET` | `/sessions/:id/checkpoints` | 读取检查点列表 |
| `POST` | `/sessions/:id/checkpoints/:checkpointId/restore` | 将当前会话回滚到检查点 |
| `POST` | `/sessions/:id/checkpoints/:checkpointId/fork` | 从检查点 fork 新会话 |
| `DELETE` | `/sessions/:id` | 关闭并释放 live session |

---

## 2. 通用约定

### 2.1 成功与错误

- `POST /sessions/:id/messages` 和 `POST /sessions/:id/control` 成功时返回 HTTP `202`。
- `DELETE /sessions/:id` 成功时返回 HTTP `204`，没有响应体。
- 参数错误返回 HTTP `400`，响应体由 NestJS 生成，前端应展示其中的 `message` 或原始文本。
- 会话不存在返回 HTTP `404`。
- 运行中的模型、工具、审批和编辑状态通过 SSE 返回，不通过发送消息的 HTTP 响应返回。

### 2.2 会话 ID

后端生成的会话 ID 形如：

```text
session-1782920000000-1
```

前端不应解析该格式，只需按 opaque string 使用。

### 2.3 URL 编码

所有路径参数都应使用 `encodeURIComponent`：

```ts
const url = `/sessions/${encodeURIComponent(sessionId)}/events`;
```

## 3. 公共类型

### 3.1 JSON 基础类型

```ts
export type JsonObject = Record<string, unknown>;
```

### 3.2 权限模式

HTTP API 当前只接受以下权限模式：

```ts
export type PermissionMode =
  | 'default'
  | 'plan'
  | 'acceptEdits'
  | 'readOnly'
  | 'bypass';
```

说明：

| 模式 | 说明 |
| --- | --- |
| `default` | 默认模式，未预先允许的敏感操作会请求审批 |
| `plan` | 规划模式，倾向只读，阻止修改 |
| `acceptEdits` | 自动接受工作区内文件编辑 |
| `readOnly` | 只允许只读工具 |
| `bypass` | 显式绕过审批，允许所有操作 |

注意：runtime 内部存在 `workspaceWrite`，但当前 HTTP controller 未开放，前端不要展示该选项。

### 3.3 模型配置

```ts
export type ModelProtocol = 'anthropic' | 'openai';

export type ModelConfig = {
  preset?: 'claude' | 'deepseek' | 'openai' | 'custom' | string;
  protocol?: ModelProtocol;
  model?: string;
  baseURL?: string;
  apiKey?: string;
  maxTokens?: number;
  temperature?: number;
  thinking?: 'adaptive' | 'disabled';
};
```

内置 preset：

| preset | 协议 | 默认模型 | 说明 |
| --- | --- | --- | --- |
| `claude` | `anthropic` | `claude-opus-4-8` | 使用 Anthropic SDK 默认地址 |
| `deepseek` | `openai` | `deepseek-chat` | 使用 OpenAI-compatible 协议，默认 `https://api.deepseek.com` |
| `openai` | `openai` | `gpt-4o` | 使用 OpenAI SDK 默认地址 |
| `custom` | 请求体提供 | 请求体提供 | 自定义模型入口 |

`apiKey` 只在创建或恢复请求中发送一次。后端不会把它写入 snapshot、日志或 HTTP 响应。


## 5. Session 生命周期接口

## 5.1 创建会话 / Create Session

### 基本信息

- **请求方式：** `POST`
- **请求路径：** `/sessions`

### 请求参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `permissionMode` | `PermissionMode` | 否 | 初始权限模式 |
| `workspaceRoots` | `string[]` | 否 | 工作区根目录列表 |
| `rules` | `PermissionRule[]` | 否 | 初始权限规则 |
| `cwd` | `string` | 否 | 会话工作目录 |
| `model` | `ModelConfig` | 否 | 模型选择与覆盖配置 |
| `mcpServers` | `McpServersConfig` | 否 | 本会话额外 MCP 配置 |
| `lspServers` | `LspServersConfig` | 否 | 本会话额外 LSP 配置 |
| `watchWorkspace` | `boolean` | 否 | 是否监听工作区文件变更 |

### 请求示例

```json
{
  "cwd": "/Users/example/project",
  "permissionMode": "default",
  "workspaceRoots": ["/Users/example/project"],
  "watchWorkspace": true,
  "model": {
    "preset": "claude",
    "thinking": "adaptive"
  }
}
```

### 自定义模型请求示例

```json
{
  "permissionMode": "default",
  "model": {
    "preset": "custom",
    "protocol": "openai",
    "baseURL": "https://api.example.com/v1",
    "model": "custom-model-id",
    "apiKey": "sk-...",
    "maxTokens": 4096,
    "temperature": 0.2
  }
}
```

### 响应示例

```json
{
  "id": "session-1782920000000-1"
}
```

### 前端处理建议

创建成功后应立即打开 `/sessions/:id/events` SSE，再发送第一条消息。后端会 replay 已缓存事件，因此先发消息再连接也能补上历史事件，但前端优先连接 SSE 更直观。

## 5.2 恢复会话 / Restore Session

### 基本信息

- **请求方式：** `POST`
- **请求路径：** `/sessions/:id/restore`

### 请求参数

```ts
export type RestoreSessionBody = {
  model?: ModelConfig;
  mcpServers?: McpServersConfig;
  lspServers?: LspServersConfig;
  watchWorkspace?: boolean;
};
```

### 请求示例

```json
{
  "model": {
    "preset": "openai",
    "model": "gpt-4o"
  },
  "watchWorkspace": true
}
```

### 响应示例

```json
{
  "id": "session-1782920000000-1"
}
```

### 前端处理建议

该接口用于恢复已经被 dispose 或服务重启后只剩 snapshot 的会话。如果会话仍 live，后端会幂等返回同一个 `id`。

## 5.3 关闭会话 / Delete Session

### 基本信息

- **请求方式：** `DELETE`
- **请求路径：** `/sessions/:id`

### 响应

成功返回 HTTP `204`，没有响应体。

### 前端处理建议

调用成功后关闭本地 `EventSource`，清空 live 状态。会话 snapshot 会在 dispose 前持久化。

## 7. SSE 事件流

## 7.1 订阅事件 / Subscribe Events

### 基本信息

- **请求方式：** `GET`
- **请求路径：** `/sessions/:id/events`
- **协议：** Server-Sent Events (SSE)

### 关键约定

后端会为每个事件设置具名 `event:`，事件名等于 payload 的 `type`。因此前端必须逐个注册事件监听器，不能只依赖 `source.onmessage`。

```ts
const source = new EventSource(`${API_BASE_URL}/sessions/${id}/events`);

for (const type of SESSION_EVENT_TYPES) {
  source.addEventListener(type, (raw) => {
    const event = JSON.parse((raw as MessageEvent<string>).data);
    handleEvent(event);
  });
}
```

后端会先 replay 当前会话的 buffered log，再继续推送 live events。

### 事件类型列表

```ts
export const SESSION_EVENT_TYPES = [
  'mcp_status',
  'lsp_status',
  'hook_event',
  'hook_failure',
  'file_created',
  'file_changed',
  'file_deleted',
  'run_status',
  'message_delta',
  'thinking_delta',
  'message',
  'tool_call',
  'tool_result',
  'approval_required',
  'edit_proposed',
  'edit_approved',
  'edit_applied',
  'edit_rejected',
  'edit_conflict',
  'permission_decision',
  'todos_updated',
  'usage',
  'context_compacted',
  'error'
] as const;
```

## 7.2 核心事件类型

### 运行状态

```ts
type RunStatusEvent = {
  type: 'run_status';
  status: 'running' | 'completed' | 'cancelled' | 'error';
  runId: string;
};
```

前端用途：控制发送按钮、取消按钮、loading 状态、检查点刷新时机。

### 流式回复

```ts
type MessageDeltaEvent = {
  type: 'message_delta';
  runId: string;
  text: string;
};

type FinalMessageEvent = {
  type: 'message';
  runId: string;
  role: 'assistant';
  content: string;
};
```

前端用途：按 `runId` 拼接 `message_delta`，收到 `message` 后将该 run 的 assistant 消息标记为 finalized。

### Thinking

```ts
type ThinkingDeltaEvent = {
  type: 'thinking_delta';
  runId: string;
  text: string;
};
```

前端用途：展示可折叠的推理过程。如果产品不展示 thinking，可保留在高级面板或忽略。

### 工具调用

```ts
type ToolInvocationSource =
  | { type: 'mcp'; server: string }
  | { type: 'builtin' }
  | { type: 'lsp' };

type ToolCallEvent = {
  type: 'tool_call';
  runId: string;
  toolUseId: string;
  name: string;
  input: JsonObject;
  source?: ToolInvocationSource;
};

type ToolResultEvent = {
  type: 'tool_result';
  runId: string;
  toolUseId: string;
  content: string;
  isError?: boolean;
  source?: ToolInvocationSource;
};
```

前端用途：按 `toolUseId` 将调用和结果配对，默认折叠展示。

### 权限审批

```ts
type ApprovalRequiredEvent = {
  type: 'approval_required';
  runId: string;
  requestId: string;
  toolUseId: string;
  name: string;
  input: JsonObject;
  message: string;
  source?: ToolInvocationSource;
  suggestions?: PermissionRuleUpdate[];
};
```

前端用途：弹出审批 UI，用户选择 allow 或 deny 后调用 `POST /sessions/:id/control`。

### 编辑提案

```ts
type FilePatchLine = {
  kind: 'context' | 'added' | 'removed';
  oldNumber?: number;
  newNumber?: number;
  text: string;
};

type FilePatchHunk = {
  oldStart: number;
  oldLines: number;
  newStart: number;
  newLines: number;
  lines: FilePatchLine[];
};

type FilePatch = {
  path: string;
  status: 'added' | 'modified' | 'deleted';
  hunks: FilePatchHunk[];
  added: number;
  removed: number;
  conflictReason?: string;
};

type EditProposalEvent =
  | {
      type: 'edit_proposed' | 'edit_approved' | 'edit_applied';
      runId: string;
      proposalId: string;
      toolUseId?: string;
      patches: FilePatch[];
    }
  | {
      type: 'edit_rejected' | 'edit_conflict';
      runId: string;
      proposalId: string;
      toolUseId?: string;
      patches: FilePatch[];
      reason?: string;
    };
```

## 8.4 恢复检查点 / Restore Checkpoint

### 基本信息

- **请求方式：** `POST`
- **请求路径：** `/sessions/:id/checkpoints/:checkpointId/restore`

### 请求体

同 `RestoreSessionBody`。

### 响应示例

```json
{
  "id": "session-1782920000000-1",
  "checkpointId": "checkpoint-session-1782920000000-1-1782920100000-1"
}
```

恢复检查点会替换当前 live session 的内部状态，但会话 ID 不变。

## 8.5 Fork 检查点 / Fork Checkpoint

### 基本信息

- **请求方式：** `POST`
- **请求路径：** `/sessions/:id/checkpoints/:checkpointId/fork`

### 请求体

同 `RestoreSessionBody`。

### 响应示例

```json
{
  "id": "session-1782920200000-2",
  "checkpointId": "checkpoint-session-1782920000000-1-1782920100000-1"
}
```

Fork 会创建一个新会话。响应里的 `id` 是新会话 ID，`checkpointId` 是来源检查点 ID。

```ts
export async function forkCheckpoint(id: string, checkpointId: string, req?: RestoreSessionRequest): Promise<CheckpointSessionResponse>;
```

错误处理建议：

- `fetch` 非 `2xx` 时读取 `res.text()`，把状态码和后端消息一起抛出。
- SSE 断线时不要立即清空会话；`EventSource` 会自动重连。
- 收到 `run_status` 的 terminal 状态（`completed`、`cancelled`、`error`）后刷新检查点。

## 10. 最小前端闭环

前端完成以下流程即可证明接口接通：

1. 调用 `POST /sessions` 创建会话。
2. 立即打开 `GET /sessions/:id/events`。
3. 调用 `POST /sessions/:id/messages` 发送用户消息。
4. 用 `message_delta` 渲染流式 assistant 文本。
5. 用 `run_status` 更新运行状态。
6. 遇到 `approval_required` 时调用 `POST /sessions/:id/control` 回复审批。
7. 遇到 `edit_proposed` 时展示 diff，并用 `edit_decision` 审批或拒绝。
8. 运行结束后调用 `GET /sessions/:id/checkpoints` 刷新检查点。
